Detailed Description

Simulates payment processing for appointment fees (UC-Payment flow).

In the current demo, every charge always succeeds and returns a randomly generated transaction ID. Completed transactions are stored in memory only — they do not survive a server restart. A real implementation would replace processPayment with an actual payment-gateway call (Stripe, Braintree, etc.) and persist the result to the database.

PaymentResponse processPayment ( PaymentRequest request)

Processes a simulated payment and returns a transaction result.

Always succeeds in demo mode. Generates a TXN-XXXXXXXX transaction ID, stores the completed status in memory, and writes an audit entry. The 500 ms delay is intentional — it mimics a real gateway round-trip for the demo.

Parameters

Returns
Response containing the transaction ID and SUCCESS status.
